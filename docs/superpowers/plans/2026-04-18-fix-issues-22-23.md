# Fix FTS5 Escaping (#22) and HTTP 500 Status Code Mapping (#23)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix two bugs — FTS5 search crashes on punctuation characters, and Apollo query returns HTTP 500 on non-Postgres backends.

**Architecture:** Issue #22 is fixed by tokenizing FTS5 input into safe alphanumeric terms joined with implicit AND (FTS5 default), so SQLite never sees raw punctuation as query syntax. The ILIKE fallback paths are hardened with manual `%`/`_` escaping (since `Sequel.escape_like` is a dataset method, not available at module level). Issue #23 is fixed by expanding `apollo_status_code` to map known unavailability error symbols to 503, and by normalizing the `direct_query` exception path to return a typed symbol instead of a raw `e.message` string, so the route layer can distinguish backend-unavailable from unexpected server errors.

**Tech Stack:** Ruby, SQLite FTS5, Sequel, RSpec

New code:
```ruby
        def fts_search(text, limit:) # rubocop:disable Metrics/MethodLength,Metrics/AbcSize
          now = Time.now.utc.strftime('%Y-%m-%dT%H:%M:%S.%LZ')
          if text.to_s.strip.empty?
            return db[:local_knowledge]
                   .where(Sequel.lit('expires_at > ?', now))
                   .limit(limit)
                   .all
          end

          tokens = text.to_s.scan(/[\p{L}\p{N}_]+/)
          if tokens.empty?
            return ilike_search(text, now: now, limit: limit)
          end

          escaped = tokens.map { |t| %("#{t}") }.join(' ')
          db.fetch(
            'SELECT lk.* FROM local_knowledge lk ' \
            'INNER JOIN local_knowledge_fts fts ON lk.id = fts.rowid ' \
            'WHERE local_knowledge_fts MATCH ? AND lk.expires_at > ? ORDER BY fts.rank LIMIT ?',
            escaped, now, limit
          ).all
        rescue StandardError => e
          handle_exception(e, level: :debug, operation: 'apollo.local.fts_search', limit: limit, fallback: :ilike)
          ilike_search(text, now: Time.now.utc.strftime('%Y-%m-%dT%H:%M:%S.%LZ'), limit: limit)
        end

        def ilike_search(text, now:, limit:)
          safe_text = text.to_s.gsub('%', '\%').gsub('_', '\_')
          db[:local_knowledge]
            .where(Sequel.lit('expires_at > ?', now))
            .where(Sequel.ilike(:content, "%#{safe_text}%"))
            .limit(limit)
            .all
        end
```

Key changes:
- Tokens joined with space (implicit AND in FTS5) — preserves conjunctive search semantics
- Extracted `ilike_search` helper with manual `%`/`_` escaping to prevent LIKE wildcard injection (`Sequel.escape_like` is a dataset method, not a module-level method, so we escape manually)
- Both the empty-tokens branch and rescue fallback use `ilike_search`

### Task 3: Normalize `direct_query` Exceptions to Typed Symbols

**Files:**
- Modify: `lib/legion/apollo.rb:242-251`

This task addresses the root cause of issue #23: when `direct_query` calls the co-located lex-apollo runner on a non-Postgres backend, the runner throws a pgvector SQL exception. The current code catches it and returns `{ success: false, error: e.message }` — a string. The route layer maps strings to 500. By normalizing this to a symbol, the route layer can map it to 503.


New code:
```ruby
      def direct_query(payload)
        log.info do
          "Apollo query using co-located reader text_length=#{payload[:text].to_s.length} " \
            "limit=#{payload[:limit]}"
        end
        Legion::Extensions::Apollo::Runners::Knowledge.handle_query(**normalize_query_payload(payload))
```

```ruby
        def apollo_status_code(result, success_status: 200)
          return 202 if result[:success] && result[:mode] == :async
          return success_status if result[:success]

          case result[:error]
          when :no_path_available, :not_started, :local_not_started,
               :upstream_query_failed, :backend_query_failed
            503
          else
            500
          end
        rescue StandardError => e
          handle_exception(e, level: :debug, operation: :apollo_status_code)
          500
        end
```

The logic:
- Known unavailability symbols → 503 (service unavailable, retryable). `:backend_query_failed` is the new symbol emitted by `direct_query` when the co-located runner throws (e.g., pgvector SQL on SQLite).
- Everything else (unknown symbols AND string errors) → 500 (server error)
